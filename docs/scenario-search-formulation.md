# Scenario as Search: A Complete Formulation

## 1. Motivation
Most advanced reasoning procedures (for example, Monte Carlo Tree Search, policy-guided expansion, and other tree-search families) can be seen as methods for navigating from an initial state toward valuable terminal outcomes (leaf nodes). They differ in strategy, but they share one structure:

- There is an initial condition.
- There is a space of possible transitions.
- There are many reachable leaves.
- The system tries to reach leaves that optimize some objective.

A scenario is this structure viewed through the lens of changing initial conditions and parameter settings.

## 2. Core Intuition
A scenario is not a single path. A scenario is a family of possible paths generated by:

- An initial condition template
- A parameter vector that instantiates that template
- A transition model that determines what can happen next

Changing parameter values changes the reachable region of the search space, the shape of the tree, and the quality/distribution of terminal outcomes.

## 3. Formal Definition of a Scenario
Define a scenario as:

$$
\mathcal{S} = (\mathcal{X}, \Theta, T, \mathcal{A}, R, \Omega)
$$

Where:

- $\mathcal{X}$: state space (all possible intermediate reasoning/problem states)
- $\Theta$: parameter space (initial-condition and environment parameters)
- $T$: transition mechanism (deterministic or stochastic)
- $\mathcal{A}$: action/decision space used by the search policy
- $R$: reward/utility/evaluation functional
- $\Omega$: constraints (resource, safety, feasibility, domain rules)

A concrete scenario instance is produced by selecting:

$$
\theta \in \Theta
$$

and instantiating an initial state:

$$
x_0 = g(\theta)
$$

Then search unfolds from $x_0$ under $T$, generating a tree (or graph) of trajectories.

## 4. Path and Outcome Semantics
A path (trajectory) is:

$$
\tau = (x_0, a_0, x_1, a_1, \dots, x_H)
$$

A leaf node is the terminal state $x_H$ under horizon or stopping rule. Its quality is measured by:

$$
J(\tau; \theta) = R(\tau) \quad \text{subject to } \Omega
$$

Key point:

- Search methods do not discover one "true" future.
- They sample or optimize over many candidate trajectories induced by the scenario instance.

## 5. What Changes When Parameters Change
Given two parameter settings $\theta_1, \theta_2$:

- Initial conditions differ: $g(\theta_1) \neq g(\theta_2)$
- Available actions or transition probabilities may differ
- Branching behavior can change
- Feasible set under constraints can change
- Terminal utility landscape can shift

Therefore, each $\theta$ defines a distinct search problem, even under the same algorithm.

## 6. Scenario Space as a Higher-Level Search
If one scenario is a search over trajectories, then multiple scenarios define a higher-level search over parameterized problems:

$$
\text{Scenario-space optimization: } \max_{\theta \in \Theta} \; \mathbb{E}_{\tau \sim \pi, T, \theta}[J(\tau; \theta)]
$$

This creates two nested levels:

- Inner level: search within a fixed scenario instance (fixed $\theta$)
- Outer level: search or analyze across scenario instances (varying $\theta$)

## 7. Relation to MCTS, Policy Search, and Tree Search
This framing is algorithm-agnostic.

- MCTS can be used as an inner solver to explore trajectories from $x_0 = g(\theta)$.
- Policy search can prioritize which branches to expand.
- Best-first and heuristic tree search can approximate high-value leaves.

The scenario definition stays the same; only the exploration policy changes.

## 8. Modeling Guidelines for Scenario Reasoning
To model scenarios rigorously:

1. Define parameter taxonomy:
- Endogenous parameters: internal state assumptions
- Exogenous parameters: environment/context variables
- Control parameters: decision levers available to the agent

2. Separate hard constraints from soft preferences:
- Hard constraints in $\Omega$ prune infeasible branches
- Soft preferences shape $R$

3. Specify uncertainty structure:
- Aleatoric uncertainty: stochastic transitions/noise
- Epistemic uncertainty: unknown model elements

4. Define terminal semantics:
- Success/failure leaves
- Risk-sensitive leaves
- Multi-objective trade-off leaves

5. Track path quality, not only final answers:
- Trajectory consistency
- Constraint violations along path
- Decision robustness under perturbations

## 9. Scenario Representation Template (Conceptual)
A complete scenario specification should include:

- Scenario name and purpose
- Parameter vector $\theta$ with domains and priors
- Initial state constructor $g(\theta)$
- Transition rules $T$
- Action set $\mathcal{A}$
- Constraint set $\Omega$
- Utility functional $R$
- Stopping/terminal conditions
- Evaluation metrics over both paths and leaves

## 10. Why This Matters for Reasoning LMs
For reasoning language models, this perspective clarifies that:

- Reasoning traces are trajectories in a scenario-conditioned search space.
- Different prompts/context assumptions instantiate different $\theta$ values.
- Monitoring signals (for example, chain-of-thought quality, tree-of-thought structure, and insight moments) can be interpreted as statistics over trajectory search behavior.

## 11. Practical Research Consequences
This formulation supports:

- Better dataset design: sample diverse $\theta$ to cover scenario space
- Better evaluation: compare robustness across parameter shifts
- Better training objectives: optimize both leaf correctness and trajectory quality
- Better monitoring: diagnose where search fails (expansion, selection, rollout, pruning)

## 12. One-Sentence Definition
A scenario is a parameterized search problem in which initial conditions and transition rules generate a family of possible trajectories, and reasoning is the process of exploring this space to reach high-value terminal outcomes under constraints.
